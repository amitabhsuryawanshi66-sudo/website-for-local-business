# Project-Level Agent Instructions

Every future design, build, or refinement task in this repository must first read the following skill contexts to ensure "Pro Max" quality standards:

- `agent-skills/emil-design-eng/` (Motion & Design Engineering)
- `agent-skills/impeccable/` (Visual Polish & Accessibility)
- `agent-skills/taste-skill/` (Anti-Generic Design & Visual Taste)
- `agent-skills/ui-ux-pro-max/` (Design System Intelligence)
- `agent-skills/21st-dev-reference/` (Component Variation Workflow)
- `docs/SKILL_STACK_OPERATING_SYSTEM.md` (Repo Foundation)

## 🛡️ Mandatory Execution Rules

1.  **Prove the Rebuild:** Never claim an architecture-level rebuild unless files are moved and `src/v2/` (or equivalent) is used.
2.  **No Ghost Redesigns:** Do not claim a redesign if only docs or CSS tokens were edited. Visible UI changes must be evident in the code and component structure.
3.  **V2 Integrity:** A version is only "V2" if `src/App.jsx` imports from the V2 folder system.
4.  **Component Cleanliness:** Do not leave old, visible components active when asked for a new master demo or rebuild. Archive them to `src/archive-v1/`.
5.  **Verified Delivery:** Always provide a list of changed files and proof of imports after any build task.
6.  **Actual Skill Application:** Use the installed skills as actionable project instructions (e.g., apply cubic-bezier easing, use bento grids), not merely as buzzwords.
7.  **Production Ready:** You must run `npm run build` before finishing any task to ensure zero errors.
8.  **Android-First QA:** Specifically audit and optimize for 360px, 390px, and 412px widths. The site must look better on a phone than a desktop.
9.  **Anti-Slop Design:** Avoid generic AI/SaaS/Tailwind-default templates. Prioritize asymmetric, human-designed compositions.
10. **Conversion Priority:** Every section must lead to a high-intent WhatsApp enquiry with a context-aware prefilled message.

## 📝 Future Task Protocol

1.  Read this `AGENTS.md`.
2.  Read the `agent-skills/` folders.
3.  Analyze the current code architecture.
4.  Categorize the task (Polish, Config, Archetype, or Rebuild).
5.  Execute using the **21st.dev-style variation workflow** for all major components.
6.  Perform an **Impeccable** polish pass before submission.
